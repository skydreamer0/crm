# CRM 醫院與產品細分改版 Roadmap

## 目標
- 把首頁上方的設定區移到獨立設定頁。
- 讓每家醫院可以設定自己可用的產品。
- 支援你提供的名單格式，例如 `慈濟/URO/許竣凱/A`。
- 保留現有的科別預設產品邏輯，作為醫院設定不足時的 fallback。

## 現況摘要
- `src/app.py` 目前同時負責首頁、設定 API、解析 API、執行 API。
- `src/templates/index.html` 目前把設定區直接放在首頁上方。
- `src/settings_store.py` 目前只處理 CRM 連線設定、LINE token、headless。
- `src/visit_list_parser.py` 目前是以科別決定產品，沒有真正的醫院層級規則。
- `src/create_appointments.py` 目前直接吃 parser 的結果並挑選產品。
- `config/department_mapping.yaml` 是科別對產品的主要來源。
- `config/product_catalog.yaml` 是產品基本資料與描述來源。

## 需求理解
- 名單資料看起來會長得像 `醫院/科別/客戶/等級`。
- 醫院名稱可能有長短名重疊，例如 `耕莘` 與 `耕莘安康`，所以解析要偏向長字串優先。
- 表頭或說明列，例如 `醫院/科別/客戶/等級08:30`，要能自動略過。
- 同一家醫院可能沒有某些產品，所以產品選擇不能只看科別。
- 設定介面要能指定「哪家醫院可以用哪些產品」。

## 建議資料模型
- 醫院資料以醫院名稱為主鍵。
- 每家醫院可以有別名，用來做解析與比對。
- 每家醫院存一組可用產品清單，產品代碼沿用 `product_catalog.yaml` 裡的 code。
- 產品選擇順序建議是：
  - 先用科別拿到預設產品清單。
  - 再套用醫院可用產品清單做交集。
  - 再套用現有的業務規則，例如 ELI 的限制。
- 若醫院設定後結果變成沒有可用產品，要回傳明確警告，不要默默亂選。

## 改版範圍
- 新增獨立設定頁，例如 `/settings`。
- 把現在首頁上的設定區搬過去。
- 在設定頁新增醫院產品矩陣。
- 擴充設定儲存格式，包含醫院與產品關係。
- 擴充 parser，能從一列資料中辨識醫院、科別、客戶、等級。
- 擴充 automation 前的產品計算流程，讓醫院限制生效。
- 補齊對應測試。

## UI 規劃
- 首頁保留：
  - 名單輸入
  - 日期選擇
  - 解析按鈕
  - 執行按鈕
  - 結果區
- 設定頁負責：
  - CRM base URL
  - 帳號
  - 密碼
  - LINE token
  - headless
  - 醫院與產品勾選
- 醫院產品區建議支援搜尋或分段顯示，避免醫院數量一多就難維護。

## 後端規劃
- `src/settings_store.py`
  - 保留現有 CRM 設定儲存。
  - 新增醫院產品設定的讀寫與公共狀態回傳。
  - 讓空白密碼欄位可保留舊值。
- `src/app.py`
  - 新增設定頁路由。
  - 擴充 `/api/settings` 的讀寫內容。
  - 在執行前檢查必要設定是否完整。
- `src/visit_list_parser.py`
  - 新增醫院識別邏輯。
  - 解析結果要帶出醫院欄位。
  - 新增醫院層級的產品過濾邏輯。
- `src/create_appointments.py`
  - 沿用 parser 的最終產品結果。
  - 不要在這一層重新猜醫院規則。

## 解析規則
- 優先從每列第一段抓醫院。
- 第二段抓科別。
- 第三段抓客戶名稱。
- 第四段抓等級或其他備註。
- 若欄位少於四段，則用現有 heuristic 補足，但不要把表頭誤判成資料。
- 醫院別名要做正規化比對，大小寫與前後空白要忽略。

## 建議實作順序
1. 先把範例資料的解析規則定死，確認醫院與科別都能正確抓到。
2. 再把設定儲存格式擴充成可存醫院產品規則。
3. 把首頁設定區搬到新的設定頁。
4. 接上醫院產品矩陣的讀寫 API。
5. 把產品選擇流程改成「科別預設 + 醫院限制 + 現有業務規則」。
6. 最後補測試與手動驗證。

## 測試項目
- `慈濟/URO/許竣凱/A` 會解析出醫院 `慈濟`、科別 `URO`。
- `耕莘安康/URO/彭崇信/B` 不會被誤切成 `耕莘`。
- `醫院/科別/客戶/等級08:30` 會被忽略。
- 沒有醫院設定時，仍會回落到科別預設產品。
- 醫院限制生效後，產品清單會被正確縮減。
- 設定頁可儲存與讀回醫院產品規則。

## 驗收條件
- 首頁不再顯示原本的上方設定區。
- 設定頁可以調整醫院與產品對應。
- 匯入的名單資料可以正確辨識醫院。
- 醫院不支援的產品不會再被自動帶入。
- 現有 CRM 自動化流程仍可正常執行。

## 風險與注意事項
- 現有檔案有編碼殘留問題，修改時要小心不要把字串規則改壞。
- 醫院名稱可能有前綴重疊，解析必須用最長比對優先。
- 醫院產品設定如果太細，UI 會變長，所以設定頁需要搜尋或分組。
- 這次改版要盡量保持向後相容，避免舊名單完全不能跑。

## 給下一個模型的交接重點
- 先讀 `src/visit_list_parser.py`，因為醫院與產品規則核心在那裡。
- 再讀 `src/settings_store.py`，確認現在設定是怎麼存的。
- UI 先拆頁，再補資料模型，避免首頁一次塞太多東西。
- 不要先重構 Playwright 流程，先把解析與設定模型穩住。
